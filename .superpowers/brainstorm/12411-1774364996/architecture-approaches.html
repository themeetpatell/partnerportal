How should the portal be structured?

Three architectural approaches — each with different trade-offs for your 3-phase roadmap

A

Single Next.js App — Route Groups

One deployment. Partner portal at /partner/*, admin portal at /admin/*. Clerk controls who can access what. Shared components, shared API routes.

Pros

  • Simplest setup & deployment
  • Shared UI components & logic
  • Fastest to ship Phase 1

Cons

  • White-label harder to isolate in Phase 3
  • Partner & admin code lives together

Best for: teams that want to move fast and add complexity later

B

Turborepo Monorepo — Separate Apps ⭐ Recommended

Two Next.js apps (apps/partner + apps/admin) sharing common packages for DB, types, UI, and business logic. Independent deployments on Vercel.

Pros

  • Clean separation of partner & admin UX
  • Shared DB schema, types & utils
  • Phase 3 white-label = new app in monorepo
  • Independent scaling & deployment

Cons

  • More initial setup vs single app
  • Turborepo learning curve

Best for: 3-phase roadmap with white-label in Phase 3

C

Separate Services — Microservices

Partner portal (Next.js), Admin portal (Next.js), and a dedicated API service (Node/Express). Each fully independent codebase and deployment.

Pros

  • Maximum isolation
  • API service reusable for white-label & webhooks

Cons

  • Highest complexity
  • Duplicated code without shared packages
  • Overkill for Phase 1
  • 3x the DevOps overhead

Best for: large teams with dedicated frontend/backend squads
